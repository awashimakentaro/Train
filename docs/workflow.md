# Vibe Coding Workflow（Self-use App）これは一例です

## 0. 背景・動機
- 自分が日常的に使いたい
- 小さく作って長く育てる
- AI × Vibe Coding の実践ログを残す

---

## 1. 初期ゴールと制約（Ultra-light 要件定義）

### ゴール
- 毎日 or 定期的に使う
- 入力・操作が直感的
- 「考えなくても使える」

### 制約
- 個人開発（1人）
- React Native + Expo
- Backend: Supabase
- UIは最小・画面数は少なめ
- 完璧な設計は最初から目指さない

---

## 2. 仮ペルソナ（= 自分）
- 技術者
- スマホ操作は速い
- 設定画面が多いアプリは使わなくなる
- 入力は最短動線が正義

---

## 3. UI検討 Figuma

### UI AI に与えた前提
- 自分専用アプリ
- Expo / React Native で実装可能
- 状態管理はシンプル
- 将来機能追加しやすい

### UI AI へのプロンプト（例）
以下の制約でモバイルアプリのUI案を3つ出してください。

- React Native (Expo) 前提

- 自分専用アプリ

- 画面数は最小

- 片手操作しやすい

- Supabase連携を想定

- SwiftUIやWeb向けの複雑なUIは不要

### 各案について

- 画面構成

- 主な操作フロー

- なぜシンプルかを説明してください。


### 採用案と理由
- 案Bを採用
- 理由：操作回数が最小 / 実装が楽 / 拡張しやすい

---

## 4. 要件定義 v1（UI反映後）

### 機能要件
- ログイン（Supabase Auth）
- メイン機能（◯◯）
- データ保存（Supabase）
- 履歴閲覧

### 非機能要件
- オフラインでも最低限使える
- ロードが遅くならない
- バグっても自分が直せる構造

---

## 5. 設計 v1（雑でOK) これは一例

### 技術スタック
- Expo
- React Native
- Supabase
- Zustand or Context（必要最低限）

### 画面構成
- Home
- Add / Action
- History
- Settings（後回し）

### データモデル（仮）
- users
- records
- metadata

※ 実装中に変更前提

---

## 6. バイブコーディング実装

### 方針
- AIに細かい指示を出しすぎない
- 動いたらOK
- 気持ち悪いところは後で直す

### 実装ログ
- 設計を変えた理由
- 追加した機能
- 消した機能

---

## 7. 実機デバッグ（Expo Go / Xcode）

- Expo Goで即確認
- 実機でUXチェック
- 「使う気になるか？」を最重視

---

## 8. 普段使い → 改善

- 1日使って不満点メモ
- UI / フローを削る
- 設定項目は極力消す

---

## 9. 振り返り（Qiita / ポートフォリオ）

- なぜこの設計にしたか
- 何を捨てたか
- AIをどう使ったか
- 次に改善したいこと
